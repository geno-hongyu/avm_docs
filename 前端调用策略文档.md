# SeedX 0627 前端调用策略文档

本文只说明前端如何按 `0627修复版` 合约语义调用后台，不再兼容旧版 `seedx_v1` 的字段解释。

## 1. 总原则

- 链上合约是唯一事实源，后台读模型只是为查询和聚合服务。
- 交易提交成功不等于后台读模型已经更新；前端必须区分 `receipt confirmed` 和 `indexed completed`。
- 治理、毕业、质押、税费、LP 分账都按新合约字段理解，不再使用旧 `startBlock/endBlock` 语义。

## 2. 交易提交流程

所有写操作统一按下面流程处理：

1. 调用对应 `tx-params` 接口，获取 `target/calldata/requiredApprovals/balanceChecks/snapshotBlock`。
2. 先串行处理 `requiredApprovals`。
3. 钱包发送主交易。
4. 轮询 `GET /api/tx/{txHash}`，直到 `indexed = true`，再刷新业务数据。

前端不要在拿到 `txHash` 或只拿到 receipt 后就直接刷新项目详情、仓位、治理列表。

## 3. 项目创建

- 前端创建项目时必须传 0627 新字段组：
  - `feeFloorBps/feeNodeBps/feeIssuerBps/feePlatformBps`
  - `taxFloorBps/taxNodeBps/taxIssuerBps/taxPlatformBps`
- `nodeCount` 和 `nodePrice` 允许传 `0` 表示使用 scale tier 默认值；传非 0 时必须落在 tier 的 `min/max` 范围内。
- 前端应先从 `/api/config/public` 读取当前 scale tier 快照，再渲染创建表单的范围和默认值。
- 合约有 `onlyEOA` 约束；若用户使用合约钱包或聚合器地址直调，前端要展示“当前创建/交易仅支持 EOA 钱包直接调用”。

## 4. 项目详情与状态

项目状态按 0627 语义展示：

- `node_sale`：节点销售中
- `trading`：已开始交易
- `graduating`：已公告毕业，等待执行窗口
- `graduated`：毕业完成，进入外部 LP 阶段
- `launch_failed`：节点销售失败

前端不要再把“毕业公告中”和“已毕业”混成同一个状态。

## 5. 用户仓位

- 毕业后 `pledgedAmount` 和 `pledgeDebt` 以链上当前值为准，直接视为 `0`。
- 历史质押行为如果需要展示，应使用历史记录接口或事件流，不要从当前仓位字段反推。
- 节点认购上限按 `maxNodePurchaseBps` 计算：

```text
单地址最多可认购节点数 = nodeCount * maxNodePurchaseBps / 10000
```

当前测试网 scale tier 为 `10000`，即单地址最多可认购全部节点。

## 6. 报价与交易

- 买卖和质押买报价都应实时请求后台 quote 接口，不要在前端复刻曲线公式。
- 前端提交交易前应再次拉一次 quote，避免旧快照报价直接下单。
- 手续费和盈利税拆分需分别展示：
  - fee split：`feeFloorBps/feeNodeBps/feeIssuerBps/feePlatformBps`
  - tax split：`taxFloorBps/taxNodeBps/taxIssuerBps/taxPlatformBps`

## 7. 治理

### 7.1 状态机

治理提案按以下状态理解：

- `active`：投票进行中
- `pending_tally`：投票已结束，等待任何人调用结算
- `passed`：已结算通过，等待 timelock 到期执行
- `rejected`：已结算拒绝
- `executed`：已执行

### 7.2 时间字段

前端必须使用以下字段，不再读取或推导旧区块字段：

- `voteStartTime`
- `voteEndTime`
- `queuedTime`

它们全部是 Unix 秒级时间戳。

### 7.3 提案限制

- 已毕业项目禁止再发治理提案。
- 同一项目、同一 `paramId`，同一时刻只允许一个未决提案。
- 前端发起提案前，应优先读取后台返回的“可发起/不可发起原因”字段，而不是自行猜测。

### 7.4 paramId 范围

0627 合约治理参数范围为 `1..11`：

1. `buyFeeRate`
2. `sellFeeRate`
3. `profitTaxRate`
4. `feeFloorBps`
5. `feeIssuerBps`
6. `feePlatformBps`
7. `taxFloorBps`
8. `taxIssuerBps`
9. `taxPlatformBps`
10. `lpFeeCreatorBps`
11. `lpFeePlatformBps`

前端展示时应优先用后台返回的 `paramLabel/oldValueDisplay/newValueDisplay/changeSummary`，不要本地硬编码全部文案。

## 8. 毕业与公告

- 毕业分两步：
  1. `triggerGraduation()`
  2. `executeGraduation(sqrtPriceX96, tickLower, tickUpper)`
- 前端要单独展示：
  - 公告时间
  - 最早可执行时间
  - 已执行时间
  - 最近公告列表
- `executeGraduation` 在 Pancake V3 模式下需要传 `tickLower/tickUpper`；Mock adapter 虽会忽略，但前端接口文档仍按完整参数组接。

## 9. 多签与平台管理

- 当前测试网 `platformAdmin` 和 `platformPayout` 都是 `PlatformMultisig`。
- 多签最少 3 signer，阈值至少 2/3。
- 前端如果提供多签操作页，应把“提交 -> 批准 -> 自动/手动执行”视为一个完整流程，而不是单交易模型。

## 10. 文档同步约定

- `/opt/seedx/docs/frontend/前端接口文档.md`

前端联调时以服务器上的最新文档为准。
