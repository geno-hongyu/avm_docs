# SeedX 20260714 前端调用策略文档

本文面向运行部署 `seedx_20260714_bca51d94`。合约 ABI 继续使用 `SeedX_BSC_Testnet_Handoff_20260714_9db55ede` 基线，测试网地址以《SeedX_BSC测试网合约地址切换交付说明_20260714》为准；本次地址切换不要求替换 ABI。旧地址缓存必须清除，前端必须按本文件和《前端接口文档》适配。

## 1. 基础规则

- BSC Testnet `chainId = 97`。
- 普通响应为 `{code,message,data}`；列表统一读取 `data.items / total / limit / offset`。
- 金额使用十进制字符串，链上整数单位只出现在明确标注为 `Wei` 的字段中。
- 写操作必须先使用登录钱包请求 tx-params，再根据 `requiredApprovals` 授权，最后发送返回的 `target/value/calldata`。每个 `requiredApprovals[].spender` 必须与同一响应的 `target` 完全一致，不一致时前端必须拒绝唤起授权。
- 只有 `preflight.readyToSubmit = true` 才允许唤起钱包发送交易。
- 交易回执成功不等于后台已完成索引；需要继续轮询交易状态或项目 indexing-status。
- SIWE 消息必须包含并校验 `Version=1 / Chain ID=97 / Issued At / Expiration Time`。登录成功后的 Bearer 会话固定 24 小时失效，`expiresAt` 为 RFC3339 绝对时间；显式注销会提前撤销，401 时必须重新钱包签名。
- Logo 上传只接受 JPEG、PNG、GIF、WebP 且最大 2 MiB；SVG 永久拒绝，不存在可由配置重新开启的入口。

## 2. 初始化配置

页面启动先请求 `GET /api/config/public`。前端不得硬编码以下内容：

- Platform、USDTTest、Multisig、Infinity Adapter/Hook 地址。
- `platformAdmin / creationFeeAdmin / platformPayout` 三种角色。
- 启用的 `scaleTiers` 及其节点数、节点价格、`nodeSaleDuration`、治理时间和毕业时间配置。
- `projectRules/sourceBundleHash`，以及每个档位的链上 `version/curveParamsHash/curvePointsHash`。

当前配置会返回 `standard-debug` 与 `standard-prod-sim` 两个启用档位。建项页面只展示接口返回且 `enabled = true` 的档位。

若 `protocolVersion`、合约地址、`sourceBundleHash` 或曲线哈希与前端预期不一致，应阻止写操作并提示刷新部署配置，不能继续使用旧缓存。

## 3. 创建项目

调用顺序：

1. `POST /api/projects/preview`
2. `POST /api/projects/create-tx-params`
3. 按 `requiredApprovals` 检查或提交 USDTTest 授权
4. 再次请求 tx-params，确认 `preflight.readyToSubmit = true`
5. 钱包发送 `createProject`
6. 轮询 `GET /api/transactions/:hash/status`
7. 等待 `indexed = true` 后刷新项目列表和详情

创建请求没有 `nodeSaleDuration`。该值由档位决定，只能从 `scaleTierSnapshot.nodeSaleDuration` 展示。

创建请求也不能提交 `maxCreationFee/expectedTierVersion/deadline`。这三项由后台在同一 `snapshotBlock` 读取当前创建费、档位版本和区块时间后自动写入 23 字段 calldata；响应中的 `maxCreationFeeWei/expectedTierVersion/deadline` 仅用于确认本次签名边界。`deadline` 默认为快照块时间后 30 分钟，不能缓存 tx-params 跨时间复用。

`tokenSymbol` 先转大写后校验：长度 1 到 10，首字符必须为 `A-Z`，其余只能为 `A-Z/0-9`。

创建参数还必须满足合约边界：`tokenName` 按 UTF-8 字节长度为 1 到 32；`minSingleBuy >= 1`；买/卖手续费为 `100..50000 ppm` 且按 `100 ppm` 步进；盈利税为 `0..300000 ppm` 且按 `100 ppm` 步进。手续费和盈利税两套四向分账都必须各自合计 `10000 bps`、floor 不低于 `5000 bps`、issuer + platform 不超过 `2500 bps`；手续费非 floor 三项归一化后的 LP creator + platform 还必须至少为 `1000 bps`。前端应即时提示，后台 preview 和 create-tx-params 会重复校验，失败时不得继续请求钱包。

预览响应 `projection` 固定返回 Requirement 14 的 13 项测算字段：节点数、节点价、节点募集总额、开盘供应量，以及 10× 价格/供应量/市值/新增 Stable/Floor/Floor Pool/Market Pool/F-P 比例/最大质押模型下外部 Stable。`curveTableVerified=true` 表示后台已在 `curveSnapshotBlock` 对 CurveTable 的 `rawPriceAtSupply / buyCost / effectiveAreaBetween` 完成同区块核验；`projectionModel=curve_table_principal_v1` 只表示曲线本金测算，不包含项目可配置交易手续费，实际可执行金额仍必须重新调用报价接口。

遇到 `CreationFeeSlippage`、`ScaleTierChanged` 或 `TransactionExpired` 时，丢弃旧 calldata，重新请求 preview 与 create-tx-params，并重新展示费用/档位让用户确认；不得只重发原交易。

未启用 Alt Token 时，`altTokenAddress` 可传空字符串或零地址，`altTokenPerNode` 可传空字符串、`"0"` 或 `"0.0"`；后端会统一编码为零地址和 0。

如果配置 Alt Token：

- `altTokenAddress` 与 `altTokenPerNode` 必须同时有效。
- 后端按 Alt Token 实际 `decimals` 编码，不按 18 位假设。
- 节点认购可能同时需要 USDTTest 和 Alt Token 两笔授权。
- 任一余额不足时，`balanceChecks[].sufficient = false`，前端禁止继续授权或发送。

## 4. 统一写交易流程

所有 tx-params 响应都应按以下顺序处理：

1. 检查 `preflight.walletType`。要求 EOA 的操作遇到 `contract` 必须阻止。
2. 展示 `balanceChecks` 中的代币符号、decimals、当前余额和差额。
3. 先本地校验 `requiredApprovals[].spender === target`，再对 `approvalRequired = true` 的项按 `amountWei` 发送精确额度 ERC-20 `approve`；任一地址不匹配都必须中止。
4. 授权确认后重新请求 tx-params，不复用旧 calldata 或旧模拟结果。
5. 仅当 `approvalsComplete = true`、`simulationStatus = passed` 且 `readyToSubmit = true` 时发送交易。
6. `estimatedGas` 用于钱包 gas 提示，不作为成功保证。

同时检查 `nativeBalanceWei / estimatedGasCostWei / nativeBalanceSufficient`。代币余额和 allowance 全部满足但原生 BNB 不足时，`readyToSubmit` 仍为 false，前端应提示补充测试网 BNB，而不是继续唤起钱包。

后端使用当前登录钱包作为 `eth_call.from`，因此前端不要用另一个钱包发送已经生成的 calldata。

## 5. 项目状态与按钮

以接口的实时 `projectState` 为准：

| 状态 | 前端主要操作 |
|---|---|
| `node_sale` | 节点认购、到期失败标记 |
| `launch_failed` | 领取退款 |
| `trading` | 买卖、质押、节点锁定、治理、触发毕业 |
| `graduating` | 展示毕业公告和倒计时，条件满足后执行毕业 |
| `graduated` | 治理、领取/分配 Infinity LP 手续费、查看外部池仓位 |

按钮可用性必须随详情或 tx-params 模拟结果刷新，不能只根据页面进入时的状态缓存。

## 6. 节点认购

- 认购上限来自项目 `maxNodePurchaseBps` 和当前地址已认购数量。
- 节点售价、`nodeTargetStable`、`nodeTokenUnit`、`nodeSaleDeadline` 均以项目详情或链上状态为准。
- `nodeFundraiseStable` 表示已售节点对应的实际募集额，不是目标募集额。
- Alt Token 项目必须同时展示 Alt Token 每节点数量、decimals、余额和 allowance。

## 7. 交易与仓位

- 买入、卖出、质押买入先调用报价接口，再请求 action tx-params。
- 报价变化后必须重新模拟，不能按 `amount * currentPrice` 自行拼交易金额。
- 0714 合约的盈利税四向分账没有公开 getter；报价中的税分账来自后台按创建 calldata 与已执行治理事件维护的确定性读模型。后台会校验四项合计为 `10000 bps`，投影缺失或非法时直接拒绝报价，前端不得自行补默认比例。
- 买入请求必须使用字段 `cashIn`，并将 `quote.tokenOut` 扣除滑点后的正数作为 `minTokenOut`；后端拒绝缺失、零值或负值。`buy(uint256,uint256)` 的 selector 固定为 `0xd6febde8`。
- 买入、卖出和质押买入都受当前总供应量 1% 的单笔上限约束；买入报价使用与合约一致的有界反解，前端不得自行放宽。
- 质押买入调用 `pledgeBuy(pledgeToken,buyToken,maxUserPay)`。`maxUserPay` 必须为报价 `userPayStable` 加滑点后的正数，selector 固定为 `0x45bd87e8`；授权金额也必须覆盖 `maxUserPay`，不能只按瞬时报价授权。
- `pledgeCostBasisStable` 是当前质押仓位的稳定币成本；部分解质押按比例恢复到普通 Token 成本，毕业后归零。
- `nodeCostBasisTotal/nodeCostBasisClaimed` 分别表示节点锁仓累计成本与已随节点释放确认的成本；不能用普通 `costBasisStable` 代替。
- 质押历史中的 `boughtToken/floorPrice` 直接来自 `PledgeCreated`，用于展示本次新铸数量和 `F_at_pledge`；历史字段不能用当前地板价回填。
- 毕业后链上质押和债务已归零；仓位页以接口当前值为准，历史记录单独展示。
- Alt Token 的锁定、已领取、可领取数量已经按真实 decimals 返回。领取必须使用 `action=claim_alt_token`，selector 为 `0x1282cfb6`；退款和节点解锁不再夹带 Alt Token。
- `claimableUnlockedToken`、`claimableFeeStable`、`claimableTaxStable` 及 Portfolio 对应合计均为链上实时值。手续费/税费已包含尚未写入 `claimableNodeFees/claimableNodeTaxes` 的节点累计奖励，前端不得再用数据库流水自行累加。
- Worker 的数据库待领取缓存采用链上绝对值定期覆盖，仅用于后台汇总与降压；它不是前端的增量账本，前端响应字段和取值规则不变。
- Portfolio 任一项目链上读取失败时接口会返回错误，不会用 0 冒充可领取金额；前端应保留上次成功画面并提示刷新失败。
- 单项目仓位和 Portfolio 的 Claimable 字段由请求时 RPC 实时读取，不是把 Worker 的 `claimables_synced_at` 缓存直接返回给用户；该缓存只用于后台汇总逐步收敛。因此前端不需要根据缓存时间自行修正金额。
- Worker checkpoint 保存区块哈希。检测到规范链哈希变化后，后台会回退安全窗口、重建 K 线及其他链上读模型并重新扫描；重建期间前端按接口错误或索引状态提示“链上数据同步中”，不要自行拼接旧 K 线。

### 7.1 消息与 WebSocket

- `GET /api/messages` 需要 Bearer 会话，但返回的是全平台统一公开时间线；不存在个人收件箱，也没有消息 `read/unread` 接口。
- 消息列表按 `items/total/limit/offset` 和项目、类型筛选；不要在本地把同一公开消息复制成用户通知。
- WebSocket 不接受 URL query 中的 session token。需要订阅客服私有会话时使用 `Authorization: Bearer <token>` upgrade header；项目公开流可匿名订阅。
- 后台会在订阅和每次投递前重新核验会话、权限及会话归属，注销或撤权后旧连接不能继续接收私有客服数据。

## 8. 治理

治理列表 `GET /api/projects/:address/governance` 使用标准分页，只读取 `items`，不再读取旧 `proposals` 字段。

状态含义：

- `active`：仍在投票。
- `pending_tally`：投票已结束，等待计票。
- `passed`：已通过，等待 timelock 或执行。
- `rejected`：计票后拒绝，已结案。
- `executed`：已执行，已结案。

能力字段：

- `allowedParamIds`：当前项目状态下合约允许的参数。
- `recommendedParamIds`：UI 推荐展示的参数。
- `unavailableParamIds`：同类别已有未结案提案，暂不可再次发起。

Trading 允许 1 到 9；Graduated 合约允许 1 到 11，但 UI 推荐 10、11。只有完整节点可发起提案。同项目同 `paramId` 在旧提案进入 `rejected` 或 `executed` 前不能再发起。

创建提案前端只提交非负整数 `newValue`。后端按合约提前校验：参数 1/2 范围 `100..50000` 且按 100 步进，参数 3 范围 `0..300000` 且按 100 步进，参数 4/7/10/11 上限 10000，参数 5/6/8/9 上限 2500；参数 4 到 6 还必须保证推导后的毕业 LP creator + platform 至少为 1000 BPS；同时校验手续费、税费和 LP 分账关系，以及相对当前值单次最多 20% 的调整幅度（当前值为 0 时使用 hard cap 的 1%）。前端应直接展示后端错误，不要继续唤起钱包。

投票是否通过必须看计票结果，不得仅比较同意票和反对票；合约还会校验快照投票权和通过条件。

- `snapshotNodeEpoch` 固定提案创建时的节点资格代际；之后新获得节点资格的钱包不能参与旧提案。
- `snapshotTotalVotingPower` 是计票使用的总投票权快照。
- 投票期内允许改票；同一钱包新的 `VoteCast` 替换旧方向和旧权重，前端不得把多次事件累加成多票。

## 9. 毕业与 Infinity

毕业流程：

1. `trigger_graduation` 生成 `triggerGraduation()`。
2. 展示公告的 `announceTime / executableAt / rollingVolume / turnoverRateBps`。
3. 公告期结束后请求 `execute_graduation`。
4. 后端只生成无参 `executeGraduation()`。
5. 索引完成后刷新 `/pool`，读取真实 `graduatedAt` 和 `infinityPosition`。

`execute_graduation` 仅在 `projectState=graduating` 时可请求，calldata 必须等于 `0x30e033fe`。`0x3dac97b1` 属于已删除的旧三参数签名，前端不得缓存或发送。

前端不得提交 `sqrtPriceX96/tickLower/tickUpper`。这些字段只会作为 `InfinityGraduated` 的结果出现在 `infinityPosition` 中。

`infinityPosition.stableDust` 是建池后剩余的稳定币尾差，首次 `collectAndDistribute()` 时会连同 LP 手续费一起分账并清零；前端不得把它当作永久未迁移资产重复累计。

`graduationWindowVolume` 是当前毕业窗口成交量；`graduationVolumeWindow` 是窗口秒数。不要再使用 `volume30d` 作为 API 字段名。

## 10. 后台管理

- 登录成功后第一步调用 `GET /api/admin/meta`。菜单、路由和操作按钮必须按 `currentAdmin.permissionKeys` 判断，不要仅按 `roleKeys` 写死，因为同一钱包可叠加多个角色。
- `meta.enums` 是管理端状态中文和操作提示的首选来源。`strict=true` 表示请求只能使用返回的值；`strict=false` 只是推荐值，前端可展示但不要当成后端硬限制。
- `GET /api/admin/platform`：读取链上平台角色、两套档位和 Infinity 配置。
- `POST /api/admin/platform/scale-tier-tx-params`：仅 `platformAdmin` 钱包可生成并模拟 19 字段 `setScaleTier`；`curvePointsHash` 必须等于目标 CurveTable 的 `pointsHash()`。
- `POST /api/admin/platform/creation-fee-tx-params`：生成 Multisig `submit` 包装交易。
- `GET /api/admin/multisig`：交易列表来自数据库分页，链上核验 signer、threshold 和 `configVersion`。交易的 `stale=true` 时禁止继续批准或执行。
- `POST /api/admin/multisig/tx-params`：支持 `submit/approve/execute/add_signer/remove_signer/replace_signer/update_threshold`；配置变更通过多签自调用执行，并会使旧配置版本下未执行交易失效。
- `POST /api/admin/configs`：调整积分倍率时，`configKey` 使用 `points.scale_tier_multiplier.<bytes32 scaleTierId>`，`configValueJson` 使用 `{"multiplier":"1.5"}`；`effectiveAt` 之后的新 Buy/PledgeBuy 积分采用新倍率。
- 平台公告必须走两步授权：先调用 `POST /api/admin/platform-announcements/authorization-message`，由当前 PlatformMultisig signer 对返回的 message 执行 `personal_sign`；收集达到当前 threshold 的不同 signer 签名后，再把同一 `title/body/deadline` 和 `approvals` 提交到发布接口。
- 管理端所有列表继续按 `items/total/limit/offset` 读取。

### 10.1 页面初始化与权限

1. 仅通过 `GET /api/auth/nonce` 与 `POST /api/auth/login` 完成 SIWE 钱包签名登录；不存在管理员用户名/密码登录接口。登录钱包仍需具备后台 RBAC 角色。
2. 调用 `/api/admin/meta`，若 401 则返回登录页，若 403 则展示“当前钱包不是后台管理员”。
3. 缓存 `protocolVersion/currentAdmin/enums/conventions/actions`，用 `permissionKeys` 构建菜单。
4. 只请求当前用户有权查看的模块；不要先请求全部接口再靠 403 隐藏。
5. RBAC 变更后立即重新请求 `meta`，不要继续使用旧权限缓存。

权限控制是前后端双层的：前端隐藏按钮只改善体验，后端仍会逐接口校验。`admin.read` 不代表拥有 `project.read/member.read/support.read/audit.read/treasury.read`。

### 10.2 列表、筛选与大数

- 页码转换：`offset=(page-1)*limit`；翻页后使用响应中的 `total` 计算总页数。
- 修改筛选、排序或 `limit` 时必须把 `offset` 重置为 0。
- 项目排序只用 `updated_at/created_at/volume_stable/transaction_count/holder_count/open_debt_stable/fdv`。
- 用户排序只用 `updated_at/created_at/point_balance/transaction_count/open_debt_stable`。
- 交易排序只用 `block_number/created_at`；`sortOrder` 只用 `asc/desc`。
- 金额、Token、价格、供应量和链上整数按字符串保存和计算，禁止直接转 JavaScript Number。
- `BPS` 的 100% 是 10000；`PPM` 的 100% 是 1000000。编辑表单显示百分比时要做单位换算，提交时再转回整数。

### 10.3 项目与用户运营标签

`POST /api/admin/targets/:targetType/:targetId/metadata` 的 `status` 只允许空字符串、`normal/watch/restricted/blocked`，`riskLevel` 只允许空字符串、`low/medium/high/critical`。两者都是内部运营标签：

- `watch/restricted/blocked` 不会自动暂停钱包、项目或链上交易。
- 真正需要公开风险提示或入口限制时，另调灰黑名单接口并填写 `reason/judgmentRule`。
- 真正需要改变链上行为时，必须走合约允许的管理员或治理交易。
- 更新成功后刷新目标详情和相应列表，确保筛选标签立即一致。

### 10.4 高风险写操作

- `scale-tier-tx-params` 和 `creation-fee-tx-params` 返回成功只代表 calldata 已生成并通过预检，不代表链上配置已改变。前端必须继续完成钱包签名、等待回执，并重新读取 `/api/admin/platform` 核验。
- 创建费更新是多签流程：先 `submit`，再由足够 signer 审批，最后执行；不能在 submit 成功后显示“创建费已更新”。
- RBAC 授予/撤销会影响下一次请求，提交前展示目标地址、角色和权限明细并二次确认。
- 灰黑名单会进入公开接口，提交前展示 `level/reason/judgmentRule`；删除后重新拉取列表和公开规则。
- 平台公告只有最终发布 API 成功才进入全体消息时间线；单独取得授权消息或收集部分签名均不代表已发布。授权 `deadline` 必须在未来 15 分钟内，signer/threshold/configVersion 变化、重复 signer、签名重放或签名不足都会被拒绝。

### 10.5 工单状态机

- `open`：待处理；可转 `pending/resolved/closed`。
- `pending`：等待反馈；可转 `open/resolved/closed`。
- `resolved`：已解决；可转 `open/closed`。
- `closed`：已关闭；只能转回 `open`。

非法流转会由后端拒绝。状态更新、回复和分配成功后，前端同时更新当前详情、列表统计和未读数；WebSocket 用于实时刷新，但 HTTP 成功响应仍是操作确认依据。

## 11. 错误处理

- 401：登录会话失效，重新签名登录。
- 403：后台 RBAC 权限不足。
- `blocked_by_approval`：先授权并等待授权交易确认。
- `insufficient_balance`：显示 `shortfallWei` 对应的可读差额，不允许继续。
- `contract_wallet_not_allowed`：当前写操作受合约 `onlyEOA` 限制。
- `StaleTransaction`：多签 signer/threshold 已变化，旧交易不能继续，必须按当前配置重新 submit。
- `CreationFeeSlippage/ScaleTierChanged/TransactionExpired`：丢弃旧建项 calldata，刷新档位和创建费后重新确认。
- `InvariantViolation 2/3/5/7`：分别表示 Floor Pool、储备负债、供应分类或质押比例核心不变式失败；停止发送并展示具体后端原因。
- `AltTokenTransferMismatch`：Alt Token 释放余额变化不精确；不要归类为余额不足或核心不变式。
- 模拟失败：直接展示后端解析后的自定义错误，不发送交易。
