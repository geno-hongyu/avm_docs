# SeedX 前端调用策略文档

> 本文档专注于**非常规逻辑**：链上交易后的刷新策略、后台接口调用时机、轮询与 WebSocket 订阅策略、错误重试策略。  
> 常规的增删改查调用流程请参阅《前端接口文档》。

---

## 一、核心架构理解

前端需要理解后台数据的两条来源路径：

```
链上确认
  └─► Worker 扫描 (6块确认 + ~30秒周期)
         └─► MySQL 读模型更新
                └─► API 返回最新数据

钱包发送 tx
  └─► receipt 到达 = "confirmed"（状态正确但后台读模型未必更新）
  └─► indexed = true = "indexed"（后台读模型已更新，可信任）
```

**关键原则**：`receipt 成功 ≠ 后台数据最新`。所有涉及链上状态的展示，必须等待 `indexed = true` 后才刷新业务数据。

---

## 二、通用交易提交流程

所有合约写操作遵循统一的六步流程：

```
① 调用 tx-params 接口 → 拿到 calldata/requiredApprovals/balanceChecks
② 本地检查余额（balanceChecks）
③ 依次发送 ERC20 approve 交易（requiredApprovals），等每笔 approve receipt
④ 发送主交易（target + calldata），拿到 txHash
⑤ 轮询 GET /api/tx/{hash} 直到 indexed = true 或超时
⑥ 刷新对应页面数据
```

### requiredApprovals 处理

```javascript
// requiredApprovals 是数组，按顺序串行处理
for (const approval of data.requiredApprovals) {
  // 先检查已有 allowance，若已满足则跳过
  const allowance = await token.allowance(userAddr, approval.spender)
  if (allowance >= BigInt(approval.amountWei)) continue
  const tx = await token.approve(approval.spender, approval.amountWei)
  await tx.wait()  // 必须等 receipt，不可并行
}
```

### snapshotBlock 的使用

`tx-params` 接口返回的 `snapshotBlock` 代表后台读取数据时的区块高度。

- **报价有效窗口**：建议 `snapshotBlock` 距当前块超过 **50 块**时提示用户重新报价
- **滑点保护**：买/卖操作中的 `minTokenOut`/`minCashOut` 基于该快照计算，超时失效的交易链上会因滑点 revert

## 三、交易状态轮询策略

### 接口
```
GET /api/tx/{txHash}
```

### 状态字段含义

| `status` 值 | `indexed` | 含义 | 前端处理 |
|------------|-----------|------|---------|
| `"unknown"` | false | 尚未在 RPC 中找到 | 继续轮询 |
| `"confirmed"` | false | receipt 已到，Worker 未索引 | 继续轮询 |
| `"confirmed_no_indexed_events"` | false | Worker checkpoint 已过此块但无事件 | 视为成功，刷新数据 |
| `"indexed"` | true | Worker 已处理，读模型已更新 | **停止轮询，刷新数据** |
| `"reverted"` (`receiptStatus`) | — | 链上 revert | 停止轮询，显示失败原因 |

### 推荐轮询参数

```javascript
const pollTxStatus = async (txHash, onSuccess, onFail) => {
  const MAX_ATTEMPTS = 30        // 最多轮询 30 次
  const BASE_INTERVAL = 3000     // 首次 3秒
  const MAX_INTERVAL = 10000     // 最长 10秒
  let interval = BASE_INTERVAL
  let attempts = 0

  while (attempts < MAX_ATTEMPTS) {
    await sleep(interval)
    const res = await api.get(`/api/tx/${txHash}`)
    const d = res.data

    // 链上 revert
    if (d.receiptStatus === 'reverted') {
      onFail(d.failureReason || '交易链上执行失败')
      return
    }
    // Worker 已索引
    if (d.indexed) {
      onSuccess(d.events)
      return
    }
    // Worker checkpoint 已过此块（无业务事件的交易）
    if (d.status === 'confirmed_no_indexed_events') {
      onSuccess([])
      return
    }

    attempts++
    interval = Math.min(interval * 1.5, MAX_INTERVAL)  // 指数退避
  }

  // 超时：receipt 已到但 Worker 未索引，给出提示而不报错
  if (attempts >= MAX_ATTEMPTS) {
    onSuccess([], { warning: '交易已确认，数据同步中，请稍后刷新' })
  }
}
```

### 状态展示建议

```
用户点击发送
  → "等待钱包确认..." (approve 阶段)
  → "交易发送成功，等待链上确认..." (txHash 拿到后)
  → "链上已确认，数据同步中..." (status = confirmed)
  → "完成！" (indexed = true)
```

## 四、各操作后的刷新策略

### 4.1 买入 / 卖出 / 质押买

**轮询完成后刷新**：

```javascript
// 刷新顺序：先价格/状态，后仓位（避免仓位先变引起用户困惑）
await Promise.all([
  refreshProjectOnChainState(projectAddr),   // GET /api/projects/:addr/state
  refreshProjectKline(projectAddr),           // GET /api/projects/:addr/kline
])
await refreshUserPosition(projectAddr, userAddr)  // GET /api/users/:addr/position/:proj
```

**不要**：
- 在 txHash 返回后立即刷新（数据未变）
- 在 receipt 到达后立即刷新（Worker 可能未处理）

### 4.2 节点认购 (subscribeNode)

```javascript
// indexed 后刷新
await refreshProjectOnChainState(projectAddr)  // nodeSold 变化
await refreshUserPosition(projectAddr, userAddr) // nodeSoldBy 变化
// 若 nodeSold === nodeCount，项目进入 Trading，需要刷新完整 runtime
if (newState.projectState === 'Trading') {
  await refreshProjectRuntime(projectAddr)
}
```

### 4.3 创建项目 (createProject)

创建项目比较特殊：**项目地址从 receipt 事件中获取**，不是从后台接口获取。

```javascript
// ① tx-params 接口只暂存 metadata（Logo/描述），不返回项目地址
// ② 发送交易后，从 receipt.logs 解析 ProjectCreated 事件获取项目地址
const projectAddr = parseProjectCreatedEvent(receipt.logs)

// ③ 轮询 tx status 直到 indexed，Worker 会自动将 logoUrl 等 metadata 绑定到项目
await pollTxStatus(txHash, () => {
  router.push(`/project/${projectAddr}`)  // 跳转到项目页
})

// ④ 进入项目页后，若接口返回 404（Worker 还未写入），最多重试 5 次（每次 3 秒）
let retries = 0
while (retries < 5) {
  const project = await api.get(`/api/projects/${projectAddr}`)
  if (project.code === 0) break
  await sleep(3000)
  retries++
}
```

### 4.4 领取解锁 / 领取手续费 / 领取税收

```javascript
// indexed 后只需刷新用户仓位，不需要刷新项目全量数据
await refreshUserPosition(projectAddr, userAddr)
```

### 4.5 治理操作（提案/投票/计票/执行）

```javascript
// indexed 后刷新治理数据
await refreshProjectGovernance(projectAddr)
// 执行提案会修改费率，需同步刷新 onChainState 获取新费率
if (action === 'execute_proposal') {
  await refreshProjectOnChainState(projectAddr)
}
```

#### 治理提案参数变更展示说明

当展示某笔治理提案（如调整费率）的变更详情时，提案属性已经由后端进行了易读化处理，前端**无需自行做单位换算（如 ppm / bps 换算），也无需硬编码映射参数名称**，直接展示下列字段即可：

- **被调整的参数名**：直接使用 `paramLabel`（如 `"买入手续费"`、`"卖出手续费"`、`"盈利税"` 等）。
- **调整前后的数值表示**：
  - 原来数值：使用 `oldValueDisplay`（例如 `"1.5%"`）。
  - 提案目标值：使用 `newValueDisplay`（例如 `"2.0%"`）。
- **一句话变更说明**：使用 `changeSummary`（例如 `"买入手续费 1.5% -> 2.0%"`）。

### 4.6 触发毕业 / 执行毕业

```javascript
// 这两步操作后刷新完整 runtime（状态机变化）
await refreshProjectRuntime(projectAddr)
// 执行毕业后提示质押用户其仓位已被销毁
if (action === 'execute_graduation') {
  showWarning('您的质押仓位已随毕业销毁，债务已清零。请检查您的仓位。')
  await refreshUserPosition(projectAddr, userAddr)
}
```

### 4.7 解除质押 (unpledge)

```javascript
// indexed 后刷新仓位和项目状态
await refreshUserPosition(projectAddr, userAddr)
await refreshProjectOnChainState(projectAddr)  // supplyPledged 变化
```

## 五、WebSocket 实时数据策略

### 订阅与退订

```javascript
// 进入项目页时订阅
ws.send(JSON.stringify({ action: 'subscribe', project: projectAddr }))

// 离开项目页时退订（路由切换钩子）
ws.send(JSON.stringify({ action: 'unsubscribe', project: projectAddr }))
```

### WS 推送的使用策略

WS 推送用于**实时 UI 动效**，不作为业务数据的权威来源：

| 推送类型 | 用途 | 不应用于 |
|---------|------|---------|
| 价格变化 | 更新价格显示、触发曲线动画 | 更新用户仓位 |
| 成交事件 | 显示最新成交 ticker | 作为余额刷新依据 |
| K 线 tick | 更新 K 线图最新 bar | — |

**WS 断连处理**：

```javascript
ws.onclose = () => {
  // 降级为轮询模式
  startPollingProjectState(projectAddr, 15000)  // 每 15 秒轮询一次
}
ws.onopen = () => {
  stopPollingProjectState()
  ws.send(JSON.stringify({ action: 'subscribe', project: projectAddr }))
  // 重连后立即刷新一次，补齐断连期间的变化
  refreshProjectOnChainState(projectAddr)
}
```

## 六、报价接口调用策略

### Quote 接口

```
POST /api/quote/buy     → 买入预估 tokenOut
POST /api/quote/sell    → 卖出预估 cashOut
POST /api/quote/pledge  → 质押买预估 userPay
```

**调用时机**：

```javascript
// 用户输入金额后 debounce 300ms 调用
const fetchQuote = debounce(async (amount) => {
  if (!amount || amount === '0') return
  const quote = await api.post('/api/quote/buy', {
    projectAddress: addr,
    cashIn: amount
  })
  setEstimatedTokenOut(quote.data.tokenOut)
  setFeeDisplay(quote.data.fee)
}, 300)
```

**Quote 失效处理**：

- 若 RPC 失败，Quote 接口直接返回错误（后台不使用本地公式）
- 前端应在 Quote 失败时禁用交易按钮，显示"报价加载中，请稍后重试"
- 发送交易前应再次调用 Quote 验证，避免报价过期

**滑点计算**：

```javascript
// 建议给用户 0.5% ~ 2% 的滑点选择
const minTokenOut = BigInt(quote.tokenOut) * BigInt(1000 - slippageBps) / 1000n
// 传给 action-tx-params 接口
body.minTokenOut = formatUnits(minTokenOut, 18)
```

## 七、接口错误处理策略

### HTTP 错误分类

| HTTP 状态码 | 场景 | 前端处理 |
|------------|------|---------|
| `400` | 参数错误（unknown field / 格式非法） | 显示 `message` 字段，不重试 |
| `401` | Session 过期或未登录 | 清除本地 token，跳转登录页 |
| `403` | RBAC 权限不足 | 显示"权限不足"，不重试 |
| `404` | 项目/用户不存在 | 显示空态，不重试 |
| `429` | 触发限流（100次/分钟） | 等待 1 秒后指数退避重试，最多 3 次 |
| `500` | 后台内部错误 | 等待 2 秒后重试，最多 2 次 |

### 合约错误回显

交易 revert 时，`GET /api/tx/{hash}` 的 `failureReason` 字段包含后台映射的合约错误原因（如 `SlippageExceeded`、`InsufficientReserve`）。前端建议提前准备对应的中文映射表：

```javascript

  'NothingToClaim':       '暂无可领取收益',

  'VotingClosed':         '投票期已结束',
  'NoVotingPower':        '您没有节点投票权',
```

## 八、节点锁定与线性释放 UI 策略

### 可领取量计算（仅用于显示，不用于实际发送）

后台 `GET /api/users/:address/position/:project` 直接返回 `claimableUnlockedToken`，前端**不要自行计算**线性释放量，直接使用接口值。

### 刷新时机

- 进入仓位页面时立即请求一次
- 每隔 **5 分钟**刷新一次（线性释放变化慢，无需高频）
- 用户执行 `claimUnlocked` 后，indexed 完成后立即刷新一次

### 解锁进度展示

```javascript
// 使用接口返回的字段，不要本地推算
const { nodeLockedTotal, nodeClaimedTotal, claimableUnlockedToken } = position

// 已释放比例（包含已领取）
const vestedRatio = (parseFloat(nodeClaimedTotal) + parseFloat(claimableUnlockedToken))
                    / parseFloat(nodeLockedTotal)
```

## 九、项目列表与状态轮询

### 项目状态轮询频率建议

| 项目状态 | 建议刷新频率 | 说明 |
|---------|------------|------|
| `NodeSale` | 30 秒 | 节点销售进度 |

```javascript
// 列表页不需要高频刷新，用户切换 tab 时刷新一次
router.beforeEach(() => {

    refreshProjectList()
  }
```

## 十、Session 管理策略

### Token 存储

```javascript
// 推荐 sessionStorage（页面关闭即清除）或内存，不推荐 localStorage
// 因为 Token 24 小时有效，用户每次打开应重新登录
sessionStorage.setItem('seedx_token', loginResponse.token)
```

### 自动登出检测

```javascript
// 所有 API 请求的 interceptor 中检测 401
axios.interceptors.response.use(null, (error) => {
  if (error.response?.status === 401) {
    clearSession()
    router.push('/login')
  }
  return Promise.reject(error)
})
```

### SIWE 登录注意事项

- 后台校验 SIWE 消息中的 `Domain` 和 `URI`，前端生成消息时必须使用**与后台配置完全一致**的域名
- 使用后台 `GET /api/auth/nonce` 接口获取 Nonce，不要本地生成
- Nonce **10 分钟**过期，用户在签名弹窗停留过久后需重新获取

```javascript
// 标准登录流程
const { nonce, message } = await api.post('/api/auth/nonce', { address })
// message 直接送给钱包签名，不要修改
const signature = await signer.signMessage(message)
const { token } = await api.post('/api/auth/login', {
  address, message, signature
})
```

## 十一、创建项目 Metadata 暂存机制

后台在 `create-tx-params` 时会将 `logoUrl/website/twitter/telegram/description` 暂存到数据库（按 `issuerAddress + tokenSymbol` 关联）。Worker 索引到 `ProjectCreated` 事件后自动绑定。

**前端注意**：

1. Logo 必须先通过 `POST /api/upload/image` 上传，拿到 `logoUrl` 后再填入创建参数
2. 若用户修改了 tokenSymbol，已暂存的 metadata 可能绑定错误，建议在改 symbol 时清空已暂存提示
3. 项目创建成功后，首次加载项目详情页时 metadata 可能有 1-2 个 Worker 周期（约 30-60 秒）的延迟才出现

## 十二、常见陷阱清单

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| receipt 到达后立即刷新余额 | 等 `indexed = true` 后刷新 |
| 本地计算线性释放量 | 使用接口返回的 `claimableUnlockedToken` |
| 并行发送 approve + 主交易 | 串行：approve receipt 后再发主交易 |
| Quote 失败时使用缓存值 | 禁用交易按钮，提示重新报价 |
| 创建项目后从后台查项目地址 | 从 receipt 的 `ProjectCreated` 事件解析 |
| 用 localStorage 存 session token | 用 sessionStorage 或内存 |
| WS 断连后不降级 | 断连后切换为轮询模式 |
| 忽略 `warnings` 字段 | 所有 warnings 必须向用户展示 |
| 滑点用 0（无保护） | 至少设 0.5% 最小滑点 |
| symbolAvailable=false 时直接报错 | 显示 warning，允许用户决定是否继续 |
| 治理投票权/票数在前端除以 1e18 显示 | 接口返回的 Display 字段（如 `forVotesDisplay`、`weightDisplay`、`totalVotingPowerDisplay` 等）已由后端按 `nodeTokenUnit` 换算为实际的“票数”，前端直接展示即可 |
## Governance Proposal Status UI
| `status` | UI label | Meaning |
|---|---|---|
| `active` | 投票中 | Proposal is still within voting period |
| `pending_tally` | 待结算 | Voting period ended, but `tallyVotes` has not been called on-chain |
| `passed` | 待执行 | Proposal passed after tally and is waiting for timelock completion |
| `rejected` | 未通过 | Proposal was tallied and did not pass |
| `executed` | 已执行 | Proposal has been executed |

Time field rule:

- For `GET /api/projects/:address/governance`
- For `GET /api/projects/:address/governance/summary`

the returned `startBlock` / `endBlock` should currently be treated as Unix timestamp seconds under the new SeedX contract, not as block numbers.

```javascript
const voteStartAt = proposal.startBlock
const voteEndAt = proposal.endBlock
```

Legacy compatibility:

- If `proposal.endBlock === 0`, this is older historical data and the backend did not preserve the proposal end time.
- In that case, the frontend should not render a countdown and should not assume `status === 'active'` means the proposal is still open for voting.
- Recommended fallback label: `历史提案` or `状态待校准`.
